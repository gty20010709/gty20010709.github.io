〇、缘起

前段时间在折腾过程中丢失了nginx的配置文件（/etc/nginx/），下面是找回缺失配置文件过程。

卸载

sudo apt-get remove nginx # 可删除除/etc/nginx 配置文件外的所有文件

apt-get -o DPkg::options::=–force-confmiss –reinstall install nginx

nom中的错误管理

译自：https://github.com/rust-bakery/nom/blob/main/doc/error_management.md

错误管理

nom 的错误设计考虑了多种需求：

  • 指示哪个解析器失败以及输入数据中的位置
  • 随着错误向上传递，累积更多的上下文信息
  • 具有非常低的开销，因为调用解析器通常会丢弃错误（例如：many0，alt）
  • 可以根据用户的需求进行修改，因为某些语言需要更多的信息

为了满足这些要求，nom 解析器必须返回以下结果类型：

结果要么是一个 Ok((I, O))，包含剩余的输入和解析出的值，要么是一个 Err(nom::Err<E>)，其中 E 是错误类型。nom::Err<E> 是一个枚举，因为组合子根据值可以有不同的行为。Err<E> 枚举表示解析器错误的 3 种情况：

  • Incomplete 表示解析器没有足够的数据来做决定。这可能是由 streaming 子模块中的解析器返回的，表示我们应该从文件或套接字中缓冲更多数据。complete 子模块中的解析器假设它们拥有整个输入数据，因此如果不够，它们将返回 Err::Error。当解析器返回 Incomplete 时，我们应该在缓冲区中累积更多数据（例如：从套接字中读取），然后再次调用解析器。
  • Error 是正常的解析器错误。如果 alt 组合子的子解析器返回 Error，它将尝试另一个子解析器。
  • Failure 是无法恢复的错误：如果子解析器返回 Failure，alt 组合子将不会尝试其他分

如果我们运行一个解析器并且知道它不会返回 Err::Incomplete，我们可以直接使用 finish() 方法从 Err::Error 或 Err::Failure 中提取错误类型：

如果我们使用了借用类型(a borrowed type)作为输入，比如 &[u8] 或 &str，我们可能想要将其转换为拥有的类型(an owned type)，以便将其传递到其他地方，使用 to_owned() 方法：

nom 提供了一个强大的错误系统，可以根据您的需求进行调整：如果要提高性能，您可以获取简化的错误信息，或者如果想要获得解析器应用的详细跟踪，可以获取精确的位置信息。

这个错误类型在 nom 的组合子中是完全通用的，因此在定义解析器时或直接在调用的地方，您可以精确地选择要使用的错误类型。请参阅JSON解析器的示例，了解在调用的地方选择不同错误类型的示例。

常见的错误类型

默认错误类型：nom::error::Error

这个结构包含一个 nom::error::ErrorKind，指示哪种类型的解析器遇到了错误（例如：ErrorKind::Tag 用于 tag() 组合子），以及错误的输入位置。

这种错误类型速度快，开销非常低，因此适用于被反复调用的解析器，比如在网络协议中。然而，它非常有限，不会告诉您解析器调用链，因此不足以生成用户友好的错误信息。

获取更多信息：nom::error::VerboseError

VerboseError<I> 类型会积累有关遇到错误的解析器链的更多信息：

它包含了每个解析器的输入位置和错误代码。它不会累积来自 alt 不同分支的错误，它只会包含它尝试的最后一个分支的错误。

它可以与 nom::error::context 组合子一起使用，以提供有关解析器链的信息：

如果直接打印，输出的信息并不容易阅读：

但通过查看原始输入和错误链，我们可以构建一个更用户友好的错误消息。nom::error::convert_error 函数可以构建这样的消息。

请注意，VerboseError 和 convert_error 被设计为语言错误的起点，但它们无法涵盖所有用例。因此，可能需要编写自定义的 convert_error 函数。

提升可用性：nom_locate 和 nom-supreme

这些 crate 的开发旨在提高编写 nom 解析器时的用户体验。

nom_locate

nom_locate 将输入数据包装在一个 Span 类型中，该类型可以被 nom 解析器理解。该类型提供位置信息，如行和列。

nom-supreme

nom-supreme 提供了 ErrorTree<I> 错误类型，它提供了与 VerboseError 相同的解析器错误链，但还累积了由 alt 尝试的各个分支的错误。

使用这种错误类型，您可以探索解析器尝试过的所有内容。

ParseError Trait

如果这些错误类型不够用，我们可以通过实现 ParseError<I> trait 来定义自己的错误类型。所有的 nom 组合子都是在这个 trait 上的泛型，因此我们只需要在解析器结果类型中定义它，它就会在所有地方被使用。

任何错误类型都必须实现该 trait，该 trait 要求提供构建错误的方法：

  • from_error_kind：根据输入位置和指示我们在哪个解析器中遇到错误的 ErrorKind 枚举。
  • append：允许在通过解析器树回溯时创建错误链（各种组合子将添加更多上下文）。
  • from_char：创建一个指示我们期望的字符的错误。
  • or：在像 alt 这样的组合子中，允许在不同分支之间选择错误（或累积它们）。

yaml入门

本文摘自：https://dev.to/paulasantamaria/introduction-to-yaml-125f ， 为我入门yaml的教程，我认为写得很好，摘抄至此。并加上了 Microsoft Translator 的翻译。

The first time I came across YAML was around a year ago when I use it to write OpenAPI definitions to document a RESTful API using Swagger API Documentation and, to be honest, I really hated it.
我第一次遇到 YAML 是在大约一年前，当时我使用它来编写 OpenAPI 定义，以使用 Swagger API 文档来记录 RESTful API，老实说，我真的很讨厌它。

Being a JSON “fan”, the YAML syntax felt weird and unnatural to me, so for a while, I didn’t pay any attention to it.
作为一个 JSON 的“粉丝”，YAML 语法对我来说感觉很奇怪和不自然，所以有一段时间，我没有注意它。

This changed a few months ago, when I started to get into CI/CD, since both Azure and GitLab pipelines require a YAML file to setup. So I finally decided to properly learn about YAML, and after doing some reading I found the ideas behind it fascinating.
几个月前，当我开始接触 CI/CD 时，情况发生了变化，因为 Azure 和 GitLab 管道都需要 YAML 文件才能设置。因此，我最终决定正确学习 YAML，在阅读了一些内容后，我发现它背后的想法很吸引人。

In this article I’ll cover the basics of YAML, including its main goals, basic syntax and some of its more complex features.
在本文中，我将介绍 YAML 的基础知识，包括其主要目标、基本语法和一些更复杂的功能。

Table of contents 目录

  • Introduction 介绍
  • Basic Syntax 基本语法
  • Distinctive Features 特色
  • Conclusion 结论

Introduction  介绍

YAML is a data-serialization language often used for configuration files, such as Open API specifications or CI/CD pipelines.
YAML 是一种数据序列化语言，通常用于配置文件，例如 Open API 规范或 CI/CD 管道。

Fun fact! 🤓 有趣的事实！🤓

According to YAML 1.0 specification document (2001-05-26) the acronym “YAML” stands for “Yet Another Markup Language”, but it was later changed to the recursive acronym “YAML Ain’t Markup Language” in the 2002-04-07 specification.

根据 YAML 1.0 规范文档 （2001-05-26），首字母缩略词“YAML”代表“Yet Another Markup Language”，但后来在 2002-04-07 规范中改为递归首字母缩略词“YAML Ain’t Markup Language”。

As stated in the latest spec YAML is designed to be friendly to people working with data and achieves “unique cleanness” by minimizing the use of structural characters, allowing the data to appear in a natural and meaningful way.
正如最新规范所述，YAML 旨在对处理数据的人友好，并通过最大限度地减少结构字符的使用来实现“独特的清洁度”，允许数据以自然和有意义的方式显示。

The latest spec also states that YAML 1.2 is in compliance with JSON as an official subset, meaning that most JSON documents can be parsed to YAML.
最新的规范还指出，YAML 1.2 作为官方子集与 JSON 兼容，这意味着大多数 JSON 文档都可以解析为 YAML。

YAML achieves easy inspection of data’s structures by using indentation-based scoping (similar to Python).
YAML 通过使用基于缩进的范围（类似于 Python）轻松检查数据结构。

Another fun fact! 🤓 另一个有趣的事实！🤓

DEV.to articles use YAML to define custom variables like title, description, tags, etc.

DEV.to 文章使用 YAML 来定义自定义变量，如标题、描述、标签等。

Basic Syntax  基本语法

YAML documents are basically a collection of key-value pairs where the value can be as simple as a string or as complex as a tree.
YAML 文档基本上是键值对的集合，其中的值可以像字符串一样简单，也可以像树一样复杂。
Here are a few notes about YAML syntax:
以下是有关 YAML 语法的一些说明：

  • Indentation is used to denote structure. Tabs are not allowed and the amount of whitespace doesn’t matter as long as the child node is more indented than the parent.
    缩进用于表示结构。不允许使用制表符，只要子节点的缩进量比父节点的缩进更多，空格的数量就无关紧要。
  • UTF-8, UTF-16 and UTF-32 encodings are allowed.
    允许使用 UTF-8、UTF-16 和 UTF-32 编码。

Conclusion  结论

Reading and writing about YAML, and experimenting with it was super interesting.
阅读和撰写有关 YAML 的文章，并对其进行实验非常有趣。

What I like: I specially loved to read about the goals of YAML in relation to code cleanness and readability, and how it achieves that. I also feel better about properly learning the syntax at last 😅.
我喜欢什么：我特别喜欢阅读 YAML 在代码清洁度和可读性方面的目标，以及它是如何实现的。最后，我也对正确学习语法😅感觉更好。

What I don’t like: I don’t like that I need a parser (which means installing a new dependency) to use YAML with the main technologies I work with (node.js and .NET Core).
我不喜欢的地方：我不喜欢我需要一个解析器（这意味着安装新的依赖项）才能将 YAML 与我使用的主要技术（node.js 和 .NET Core）一起使用。

However, I will now consider YAML, specially if I need something that JSON can’t cover like reusability, explicit types or comments. I’m sure that working with pipelines will be easier now too.
但是，我现在将考虑 YAML，特别是如果我需要 JSON 无法涵盖的东西，例如可重用性、显式类型或注释。我敢肯定，现在使用管道也会更容易。
